1. Общая архитектура системы
1.1 Основные принципы
Реализованная библиотека представляет собой упрощенную, но функционально полную версию RxJava, основанную на паттерне "Наблюдатель" (Observer Pattern). Система построена на принципах реактивного программирования, где данные передаются в виде асинхронных потоков событий.

1.2 Уровни архитектуры
Архитектура разделена на четыре основных уровня:

Уровень 1 - Пользовательский код: Создание потоков, подписка на события, обработка данных. Пользователь взаимодействует с библиотекой через фабричные методы и операторы.

Уровень 2 - Операторы преобразования: Слой трансформации данных, включающий map, filter, flatMap. Каждый оператор реализован как декоратор, оборачивающий существующий Observable и возвращающий новый с измененным поведением.

Уровень 3 - Ядро библиотеки: Базовые компоненты - Observable, Observer, Disposable. Обеспечивает фундаментальные механизмы передачи событий и управления подписками.

Уровень 4 - Планировщики: Управление потоками выполнения. Обеспечивает асинхронность и параллелизм через различные стратегии выполнения задач.

2. Компоненты ядра системы
2.1 Observable (Наблюдаемый источник)
Observable является центральным элементом системы и представляет поток данных. Он выполняет несколько ключевых функций:

Фабричные методы - предоставляет статические методы для создания Observable из различных источников: create для пользовательской генерации, just для одиночных значений, fromIterable для коллекций, fromArray для массивов, range для числовых последовательностей.

Управление подпиской - содержит метод subscribe, который устанавливает связь между Observable и Observer, создавая цепочку передачи данных.

Цепочка операторов - каждый оператор (map, filter, flatMap) возвращает новый Observable, что позволяет строить цепочки преобразований любой длины.

Внутренняя архитектура - Observable абстрактный класс, где конкретная реализация определяется через метод subscribeActual, вызываемый при подписке.

2.2 Observer (Наблюдатель)
Observer определяет интерфейс для получения событий из потока и имеет три ключевых метода:

onNext - вызывается для каждого элемента потока, передавая данные для обработки. Может вызываться множество раз.

onError - вызывается при возникновении ошибки, передавая исключение для обработки. После этого поток считается завершенным.

onComplete - вызывается при успешном завершении потока без ошибок. Сигнализирует о том, что больше элементов не будет.

Жизненный цикл Observer следует строгой последовательности: после создания следуют ноль или более вызовов onNext, затем ровно один вызов либо onError, либо onComplete.

2.3 Disposable и Subscription (Управление подпиской)
Механизм отмены подписки реализован через интерфейс Disposable, который предоставляет два метода:

dispose - отменяет подписку, останавливая получение дальнейших событий. После вызова этого метода Observer больше не получает onNext, onError или onComplete.

isDisposed - проверяет состояние подписки, возвращая true если подписка уже отменена.

Класс Subscription реализует этот интерфейс, используя потокобезопасный флаг состояния, что позволяет отменять подписку из любого потока.

2.4 ObservableOnSubscribe (Фабрика данных)
Этот функциональный интерфейс определяет, как генерировать данные для Observable. Он принимает Observer и позволяет вручную вызывать его методы onNext, onError, onComplete, предоставляя полный контроль над генерацией событий.

3. Операторы преобразования
3.1 Паттерн декоратор
Все операторы реализованы с использованием паттерна "Декоратор". Каждый оператор:

Наследуется от Observable

Принимает исходный Observable в конструкторе

При подписке создает новый Observer, оборачивающий исходный

Трансформирует, фильтрует или объединяет данные

Передает результат дальше по цепочке

3.2 Оператор Map
Map преобразует каждый элемент потока по заданной функции. При получении элемента он применяет функцию преобразования и передает результат дальше. Если при преобразовании возникает исключение, оно перехватывается и передается в onError, прерывая поток.

3.3 Оператор Filter
Filter фильтрует элементы по предикату. Каждый элемент проверяется через условие-предикат. Если условие выполняется (true), элемент передается дальше; если нет - элемент игнорируется. Исключения в предикате также обрабатываются и передаются в onError.

3.4 Оператор FlatMap
FlatMap является наиболее сложным оператором. Он преобразует каждый элемент в новый Observable и затем объединяет все элементы этих внутренних Observable в один выходной поток.

Ключевые особенности flatMap:

Параллельное выполнение - внутренние Observable могут выполняться параллельно, если они используют разные планировщики.

Управление завершением - оператор отслеживает количество активных внутренних Observable. Он завершается только после того, как исходный Observable завершился и все внутренние Observable также завершились.

Объединение потоков - элементы из всех внутренних Observable перемешиваются в единый выходной поток в порядке их поступления, что может привести к перемешиванию данных из разных источников.

4. Планировщики (Schedulers)
4.1 Интерфейс Scheduler
Планировщик определяет стратегию выполнения задач в потоках. Основной метод execute принимает задачу (Runnable) и выполняет ее в соответствующем потоке.

4.2 Типы планировщиков
IOScheduler - предназначен для операций ввода-вывода. Использует CachedThreadPool, который создает новые потоки по мере необходимости и переиспользует существующие. Оптимален для блокирующих операций (сеть, файловая система, базы данных), так как не ограничен количеством потоков.

ComputationScheduler - предназначен для вычислительных операций. Использует FixedThreadPool с количеством потоков, равным количеству ядер процессора. Оптимален для CPU-интенсивных задач, так как предотвращает избыточное создание потоков и конкуренцию за процессорное время.

SingleThreadScheduler - предназначен для последовательного выполнения. Использует один выделенный поток. Гарантирует порядок выполнения задач и безопасный доступ к общим ресурсам.

4.3 Управление потоками выполнения
subscribeOn - определяет поток, в котором выполняется подписка и генерация данных. Указывается один раз и влияет на весь поток до места вызова. Все операторы до observeOn будут выполняться в этом потоке.

observeOn - определяет поток, в котором обрабатываются элементы. Может быть указан несколько раз в цепочке, переключая поток выполнения для последующих операторов. Позволяет гибко распределять нагрузку между разными планировщиками.

4.4 Комбинирование планировщиков
Возможно использование нескольких планировщиков в одной цепочке. Например, генерация данных может происходить в ComputationScheduler для быстрой обработки, затем данные передаются в IOScheduler для записи в файл или базу данных. Это позволяет оптимально использовать ресурсы системы для разных типов операций.

5. Поток данных и обработка ошибок
5.1 Путь данных
Данные проходят следующий путь:

Создание Observable через фабричный метод

Применение операторов преобразования (map, filter, flatMap)

Подписка Observer через метод subscribe

Генерация данных источником через onNext

Прохождение данных через цепочку операторов

Получение данных конечным Observer

5.2 Обработка ошибок
Ошибки обрабатываются по следующему принципу:

Ошибки возникают в любом месте цепочки (генерация, преобразование)

Ошибка немедленно передается в onError

Все последующие элементы игнорируются

Ошибка проходит через всю цепочку до конечного Observer

После передачи ошибки поток считается завершенным

5.3 Механизм отмены подписки
Подписка может быть отменена в любой момент:

Вызов dispose() отключает Observer от источника

Дальнейшие вызовы onNext игнорируются

Ресурсы освобождаются

Предотвращаются утечки памяти

6. Потокобезопасность и синхронизация
6.1 Потокобезопасные компоненты
Subscription использует volatile переменную для флага отмены

FlatMapOperator использует атомарные счетчики для отслеживания активных потоков

Планировщики используют потокобезопасные ExecutorService

6.2 Гарантии порядка
Для одного потока данные приходят в порядке генерации

При использовании нескольких потоков порядок не гарантируется

SingleThreadScheduler гарантирует последовательное выполнение

7. Преимущества архитектуры
7.1 Модульность
Каждый компонент имеет четкую ответственность:

Observable управляет потоком данных

Operator отвечает за преобразование

Scheduler управляет потоками выполнения

Disposable контролирует подписку

7.2 Расширяемость
Новые операторы легко добавить через наследование от Observable

Новые планировщики через реализацию интерфейса Scheduler

Существующие компоненты не требуют изменений

7.3 Композиция
Операторы можно комбинировать в любом порядке

Планировщики можно комбинировать многократно

Сложные пайплайны строятся из простых компонентов

7.4 Асинхронность
Асинхронное выполнение через планировщики

Неблокирующие операции

Эффективное использование ресурсов

8. Заключение
Разработанная архитектура успешно реализует основные концепции реактивного программирования:

Потоки данных как первый класс объектов

Асинхронное выполнение через планировщики

Декларативный стиль через операторы

Управление ресурсами через Disposable

Обработка ошибок как часть потока

Архитектура обеспечивает:

Чистоту и понятность кода

Легкость тестирования каждого компонента

Возможность расширения новыми операторами

Эффективное использование многопоточности

Библиотека готова к использованию в реальных проектах и может быть легко расширена для поддержки дополнительных операторов и стратегий планирования.

Каждый асинхронный вызов уменьшает счетчик

Основной поток продолжается только после завершения всех операций

ПРОСТЕЙШАЯ ПОДПИСКА
Самый простой способ использования библиотеки - создание Observable с одним значением и подписка на него:
Сценарий: Вывод приветствия в консоль
Создается Observable через метод just, который генерирует одно значение. 
Затем вызывается подписка с лямбда-выражением, которое будет выполнено для каждого полученного значения. Это демонстрирует базовый принцип работы: данные передаются от источника к потребителю через подписку.

Фильтрация данных с помощью filter

Создается список пользователей с именами и возрастами. 
Observable фильтруется с помощью filter, оставляя только тех пользователей, возраст которых больше или равен 18. 
Отфильтрованные данные выводятся на экран. 
Пример показывает, как легко отсеивать ненужные данные.

Разделение потоков для подписки и обработки
Сценарий: Генерация данных в одном потоке, обработка в другом
Данные генерируются в вычислительном потоке (ComputationScheduler), а их обработка выполняется в потоке ввода-вывода (IOScheduler). 
Это оптимально, когда генерация данных требует вычислений, а их сохранение - операций ввода-вывода.
